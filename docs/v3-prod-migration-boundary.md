# V2 → V3 Prod 数据迁移范围与运营边界

确认日期：2026-07-15

## 1. 文档状态与目标

本文件是 V2 → V3 prod 数据迁移筛选、演练、校验和正式切换的正式契约。

当前状态分为两层：

- **已确定迁移政策**：本文件中的模块范围、时间口径、引用闭包、历史身份、Cash 边界、验收指标和切换方式已经确定。
- **演练与未来执行阶段**：School 私塾打工与 Cash ledger 已完成一次受控 staging 初始副本演练；`v3-prod`、正式迁移批次、final delta / freeze 和生产切换均尚未执行。

本次文档整理不执行 SQL / RPC，不写任何数据库，不复制 V2 数据，也不修改业务代码或前端。

迁移不采用全系统统一日期截断。每个业务模块必须按自己的业务事实日期和审计链定义范围，不得用 `created_at` 代替业务月份。

## 2. 已确定的模块化迁移范围

| 模块 | 业务范围字段 | 已确定迁移范围 | 处理原则 |
| --- | --- | --- | --- |
| 普通学生预定 / 实际课时 | 课时业务日期、`year_month` | `2026-07` 及以后 | 迁移事实记录；V3 切换后由 V3 继续记录 |
| 学生月度结算 | `year_month` | `2026-07` 及以后 | 不迁入更早的非依赖事实数据 |
| 学费账单 / 学费收入 | `billing_month` / `year_month` | `2026-07` 及以后 | 与课时、结转、收入和 Cash 引用保持完整 |
| 老师工资 / 工资支出 | `settlement_month` / `year_month` | `2026-07` 及以后 | 与实际课时、工资明细和支出保持完整 |
| 普通收入 / 支出 | 来源业务月 / `year_month` | 来源模块处于迁移范围时 | 不按收付款发生日单独截断 |
| 私塾打工 / 外部授课 | `year_month` | 完整“2026 结算年度”，即 `2025-12` 至 `2026-11` | 迁移完整事实与审计链，不只迁移月度汇总 |
| 主数据 | 被事实记录引用的 ID | 按引用闭包 | 不按创建日期裁剪 |
| Cash 引用 | 跟随 School 来源链 | 来源模块处于迁移范围时 | 是否复制 Cash ledger 取决于第 8 节的部署边界 |

普通教学业务的正式运营起点仍为 `2026-07`。这个边界来自 2026 年 7 月普通教学业务归属规则收口：V3 新业务不再让操作员选择历史多业务归属，而是使用固定运营归属，因此不把收口前的历史多归属事实链带入 V3 prod。私塾打工属于另一套按工作单位结算的独立系统，不使用普通教学业务归属维度，所以按完整 2026 结算年度迁移。该年度范围不得扩张为普通教学业务的历史全量迁移。

## 3. 普通教学业务政策

普通教学业务包括学生课程、学生月度结算、学费账单、老师工资及其正式下游财务记录。

已确定政策：

- 原则上只迁移业务月 / 结算月为 `2026-07` 及以后的正式业务事实。
- `2026-07` 是普通教学业务归属规则的收口点，不是对所有 V3 模块生效的统一日期截断。
- 收口前涉及历史个人名义 / 多业务归属的普通教学事实默认留在 V2 只读系统，避免把已经取消的新业务选择维度和历史归属分支带入 V3 prod。
- V2 已统一的 2026 年 7 月预定 / 实际课时迁入 V3 后，尚未在 V2 生成的月结、账单、工资、收入、支出和 Cash 链路优先由 V3 原生生成。
- 如果切换前 V2 已生成迁移范围内的下游记录，必须先盘点并选择“迁移原链”或“由 V3 原生重建”中的唯一一种路径，禁止两边各生成一次。
- `2026-06` 及以前的普通教学事实默认保留在 V2 只读历史系统；只有作为迁移范围内记录的必要引用时，才随引用闭包迁入。
- 即使业务月在 `2026-07` 及以后，若某普通教学历史链依赖未进入 V3 的多维工资明细 / 调整或学生月结 adjustment / carryover，则该受影响链整体保留在 V2 只读，禁止只迁移其汇总或在 V3 重算替代值。支出附件和 legacy payment request 按第 10.2 节的已批准 omission policy 处理。
- V3 正式切换后，V2 停止对相同业务范围继续写入，不采用长期双写。

## 4. 私塾打工完整 2026 结算年度

“2026 结算年度”固定定义为：

```text
2025-12 至 2026-11
```

迁移窗口按实际切换点计算：

- 如果 V3 在 2026 年 7 月切换，首次迁移 V2 已存在的 `2025-12` 至 `2026-06` 数据；`2026-07` 至 `2026-11` 在 V3 正常记录。
- 如果实际切换时间晚于 2026 年 7 月，则迁移截至冻结 / 切换点已在 V2 形成的全部 2026 结算年度数据。
- 无论切换月份为何，最终 V3 中必须形成完整的 `2025-12` 至 `2026-11` 年度链路，且同一业务月只能有一个正式写入系统。
- V3 必须在 2026 年 12 月新结算年度开始前完成切换；切换前月份从 V2 迁移，切换后至 2026-11 的月份在 V3 正常记录。

范围判断只使用私塾打工业务月 `year_month`。不得按 `income_date`、`payment_date`、Cash 请求日期、Cash 确认日期或交易日期裁剪。

私塾打工以工作单位和 `year_month` 作为结算维度，不依赖普通教学的 `business_entity_id`。不得为了迁移方便给历史私塾打工记录补造、推断或改写业务归属。

例如：2026-04 工资在 2026-05 到账时，2026-04 结算、对应收入、联动事件和 Cash 引用属于同一条迁移链，必须整体处理。

## 5. 私塾打工审计链范围

私塾打工不得只迁移月度汇总。未来迁移清单至少必须覆盖并核对以下 V2 来源表或等价来源：

1. `school_historical_part_time_work_import_batches`
2. `school_part_time_work_lessons`，同时覆盖 `planned` / `actual`
3. `school_part_time_work_monthly_settlements`
4. `school_part_time_work_monthly_settlement_details`
5. 对应的 `school_income_records`
6. `school_personal_cash_income_linkage_events`
7. 仍被上述记录引用的 legacy income request 数据（如存在）
8. 被上述记录引用的工作单位和其他私塾打工必要主数据；不包含普通教学业务归属

V3 当前目标模型包括 `external_workplaces`、`external_work_lessons`、`external_work_monthly_settlements`、`external_work_settlement_details`、`income_records` 和 `cash_requests`。正式执行前必须建立逐表、逐字段映射，证明 V2 每条来源记录在 V3 中的去向；没有安全映射的字段不得静默丢弃。

V3 已通过版本化 schema / migration 增加历史导入批次、原始行、来源 snapshot / hash 与 legacy Cash linkage 审计承载，并已用于私塾打工 staging 初始副本演练。普通教学模块仍必须先完成逐表字段 mapping 与受控来源盘点，确认同等身份和审计字段可无损落位；本文件不授权使用备注字段拼接、名称匹配或重新生成身份来绕过该缺口。

## 6. 历史身份与不可改写字段

私塾打工迁移必须原样保留或建立一一对应、可证明无损的审计映射：

- 原 UUID；有直接 V3 对应记录时必须沿用为目标记录 ID，不得为历史事实重新生成业务身份。
- `historical_import_batch_id`。
- `historical_source_row`。
- `source_snapshot`。
- 来源文件哈希。
- planned / actual 关联关系。
- 结算状态、结算明细和收入状态。
- Cash transaction ID、external request / event ID 和相关幂等键。
- V2 来源表、来源 ID、迁移批次、迁移时间和迁移程序版本。

如果当前 V3 目标表或约束不能沿用 V2 UUID，必须在迁移前先调整版本化 schema，使原 UUID 能作为目标记录 ID 或专用历史记录的不可变主身份。legacy identity mapping 可以作为附加索引和审计证据，但不得替代原 UUID，也不得只按名称、月份或金额匹配。

`school_personal_cash_income_linkage_events` 中的 `personal` 表示历史个人 Cash 收入联动语境，不等于普通教学 `business_entity_id`。迁移必须保留该联动事件和 Cash 身份，但不得由表名推断或补造业务归属。

## 7. 主数据引用闭包

普通教学与私塾打工分别计算自己的引用闭包，不共用一套主数据假设。

迁移规则：

- 先确定各模块的事实记录范围，再递归计算所有非空外键和审计引用闭包。
- 被普通教学 `2026-07+` 事实引用的学生、老师、科目、账户、固定运营归属和规则必须迁移，即使创建时间早于 2026-07。
- 被私塾打工完整结算年度引用的工作单位、导入批次、收入和 legacy 请求 / Cash 身份必须迁移；私塾打工不引用普通教学业务归属。
- 历史主数据可以在 V3 中保持 archived / history-only，不因此开放为新业务选项。
- 每个引用必须通过 UUID 或受控 ID mapping 解析，迁移后不得存在孤立外键或用显示名称替代引用。

## 8. Cash 数据边界

Cash 边界在执行前必须二选一并记录环境拓扑，不能临时混用。

### 8.1 V3 继续使用现有 Cash DB

- School 侧只迁移 Cash 引用、外部 ID、状态和联动审计。
- 不复制、不重新生成 Cash 请求、确认事件或账户交易。
- 必须验证每个 `synced` 引用的 Cash transaction ID 在现有 Cash DB 中唯一且有效。

### 8.2 Cash 同时迁移到新数据库

- 另立独立的 Cash ledger 迁移阶段和审批文件。
- 按账户流水、请求、事件、交易和幂等关系迁移完整账本。
- Cash ledger 迁移不得与本 School 数据移行阶段混做。

任何路径都禁止因为 School 数据移行而产生重复 Cash 流水。历史确认记录不得伪造新的 Cash request / event / transaction 身份。

## 9. 迁移一致性与幂等要求

- 迁移程序必须幂等；同一来源记录和迁移批次重跑不得生成第二条业务记录或第二笔 Cash 引用。
- 每类记录必须保存来源系统、来源表、原 ID、迁移批次和内容校验值。
- 迁移顺序必须先依赖主数据，再事实记录、明细、结算、收入、联动事件，最后验证 Cash 引用。
- 不使用当前汇率反向重算历史 JPY / CNY 金额，不用当前业务规则重新解释历史 snapshot。
- 不用 `created_at`、到账日或交易日推断私塾打工业务月。
- 任何差异必须进入异常清单；不得在 prod 手工拼接半条审计链。

## 10. 验收口径

### 10.1 私塾打工业务对账

按工作单位、业务月逐项核对：

- planned / actual 课时数量。
- 结算课时数和总课时。
- 课时工资。
- 交通费。
- 调整项。
- JPY 总额。
- 实际到账 CNY。

同时核对：

- 导入批次、planned / actual、结算、结算明细、收入和联动事件的记录数量。
- planned → actual → settlement detail → settlement → income → linkage / Cash reference 的引用完整性。
- 原 UUID、批次 ID、原始行、snapshot、文件哈希和状态无损保留。
- 历史确认记录没有伪造或重建 Cash 身份。
- 每个 `synced` 记录引用的 Cash transaction ID 唯一且有效。
- legacy income request 只在仍被迁移链引用时保留，且不存在孤立请求。

### 10.2 普通教学业务对账

- 按学生、老师、科目和业务月核对 `2026-07+` 预定 / 实际课时数量、时长和金额。
- 验证学生结算、学费账单、工资、收入和支出只读取迁移范围内来源。
- 核对受控 snapshot 的 exclusion manifest：含工资明细 / 调整或学生月结 adjustment / carryover 的每条受影响链均保留在 V2 只读；附件未复制；legacy payment request 未创建 V3 `cash_requests`。
- 验证没有混入 `2026-07` 以前的非依赖普通教学事实数据。
- 验证被保留事实引用的主数据完整，未因创建日期较早而断链。
- 验证私塾打工年度例外没有扩大到普通教学模块。

### 10.3 技术验收

- 所有非空外键均可解析，孤立引用数为零。
- ID mapping 一对一且无重复来源 / 目标。
- 迁移脚本重复运行时新增业务记录和 Cash 流水均为零。
- 每个迁移批次保留清单、行数、金额摘要、校验哈希、异常、操作者和时间戳。
- staging 演练与 prod 正式导入使用同一版本迁移程序和校验程序。

## 11. 正式切换方式

正式切换采用：

```text
全量初始迁移
-> 对账与异常修正
-> 最终增量 / V2 写入冻结
-> 最终对账
-> 单点切换到 V3
```

执行步骤：

1. 建立独立 staging / prod 环境和密钥，禁止复用 dev 数据库。
2. 盘点 V2 来源、目标 schema 缺口、Cash 拓扑和字段映射。
3. 在 staging 使用受控副本完成至少一次全链路迁移与幂等重跑。
4. 备份 V2 来源和 V3 prod 初始状态，生成全量初始迁移批次。
5. 完成模块行数、金额、引用、身份和 Cash 边界对账。
6. 冻结 V2 对迁移范围的写入，导入最终增量并再次对账。
7. 单点切换日常运营入口到 V3；V2 转为只读历史查询。
8. 保留迁移报告、异常审批、回滚点和切换时间戳。

不允许长期双写。任何关键校验失败都必须停止下游生成和切换，按迁移批次回滚，不在 prod 中手工修补后继续上线。
