# V3 Prod 数据迁移与运营边界

确认日期：2026-07-13

## 1. 目标

V3 prod 以 `2026-07` 为正式运营起点，只承接青空进学塾的新运营数据，不背负 V1 / V2 的个人名义、早期实验链路和过渡期 Cash linkage 复杂度。

本文件是 V2 -> V3 prod 迁移筛选、演练、校验和上线切换的正式边界。任何扩大范围的例外都必须单独记录并审批。

当前状态：迁移边界已确认，但 staging / prod 数据库、正式迁移脚本和 prod 数据导入尚未执行。

## 2. 系统分工

- V3：承接业务月 / 结算月为 `2026-07` 及以后的正式运营主链路。
- V2：保留为历史查询、审计和对账来源，主要覆盖 `2026-06` 及以前数据和必要的过渡期查询。
- V1：只在极端情况下作为数据修正参考或工具，不进入日常运营，也不作为 V3 业务设计依据。

V3 上线后不得在 V2 和 V3 同时生成同一业务月的正式下游单据。

## 3. 业务归属边界

- V3 prod 新业务唯一运营归属为 `aozora_school`（青空进学塾）。
- `personal`（个人名义）不作为 V3 prod 新业务选项。
- V2 历史个人名义记录不全量迁移到 V3。
- V3 保留 `business_entity_id` 字段和按业务归属的历史兼容能力，但 prod 首期不开放个人名义写入。
- 学费、月结、工资、收入、支出和 Cash 请求等派生记录必须继承上游业务归属，前端不得重新选择。

## 4. 时间筛选规则

迁移不能以 `created_at` 作为业务范围判断。正式筛选字段如下：

| 数据类型 | 业务筛选字段 | 正式范围 |
| --- | --- | --- |
| 预定 / 实际课时 | `lesson_date`、`year_month` 或 V2 对应业务日期字段 | `>= 2026-07` |
| 学生月度结算 | `year_month` | `>= 2026-07` |
| 学费账单 | `billing_month` / `year_month` | `>= 2026-07` |
| 老师工资 | `settlement_month` / `year_month` | `>= 2026-07` |
| 收入 / 支出 | `business_month` / `year_month` | `>= 2026-07` |
| Cash 请求 / 回写 | 跟随来源收入或支出的业务月与业务归属 | 来源满足迁移范围时才迁移 |

所有主链路业务记录还必须满足归属青空进学塾。早期实验数据、旧 personal Cash linkage 和没有正式来源的数据不得因创建时间较晚而进入迁移范围。

## 5. 主数据引用闭包

学生、老师、科目、工资规则等主数据不能简单按创建时间过滤。只要被迁移范围内的 `2026-07+` 正式业务记录引用，就应迁移对应主数据，并建立稳定的 V2 -> V3 ID 映射。

主数据迁移原则：

- 迁移 `2026-07+` 正式课时引用的学生、老师、科目和青空进学塾工资规则。
- 迁移 V3 正式运营所需的青空进学塾业务归属和必要账户配置。
- 不因主数据创建于 2026-06 以前而遗漏其 2026-07+ 引用关系。
- 不把未被正式范围引用的历史 personal 主数据关系批量带入 V3。
- ID 映射必须可重复使用并支持幂等重跑，禁止只依赖名称匹配。

## 6. 2026 年 7 月首批迁移路径

V2 已将 2026 年 7 月预定课时和实际课时手动统一为青空进学塾；目前 7 月学生月结、老师工资、学费应收、收入和 Cash 链路尚未生成。

因此首批迁移采用以下路径：

```text
V2 青空进学塾主数据
-> V2 2026-07 预定课时 / 实际课时
-> V3 prod 主数据与课时
-> 在 V3 原生生成学生月结
-> 在 V3 原生生成学费账单 / 老师工资
-> 在 V3 原生生成收入 / 支出
-> 从 V3 发起后续 Cash 链路
```

不得在 V2 先生成 2026-07 下游单据后，又在 V3 重复生成同一批正式记录。迁移冻结后如 V2 出现新的 2026-07 下游记录，应停止导入并先做差异审计。

## 7. 明确不迁移的内容

- `2026-06` 及以前默认历史业务数据。
- V2 历史个人名义业务记录。
- V2 早期实验链路、测试数据和临时修复数据。
- 旧 personal Cash linkage、无正式来源的 Cash 请求和过渡期回写记录。
- V1 日常运营数据模型和历史页面结构。
- 仅为 V2 兼容而存在、V3 主链路不再需要的中间表或冗余字段。

如对账需要查看上述内容，应回到 V2 历史查询系统，而不是扩大 V3 prod 主链路数据范围。

## 8. 迁移一致性要求

- 每类来源记录必须保存 V2 来源表、来源 ID 和迁移批次标识。
- 迁移脚本必须幂等，同一批次重跑不能生成重复学生、课时或下游单据。
- 迁移顺序必须先主数据、后课时；7 月下游快照由 V3 业务 API / domain service 原生生成。
- Cash 请求只能跟随已经迁移并在 V3 建立的收入 / 支出，不允许孤立迁移。
- 不使用当前汇率反向重算历史金额，不用 `created_at` 推测业务月份。
- 迁移期间 V2 来源数据需要冻结，或建立明确的增量差异清单。

## 9. 校验清单

迁移演练与正式导入至少需要验证：

1. 2026-07+ 青空进学塾预定课时、实际课时数量与来源一致。
2. 每条课时的学生、老师、科目、日期、时长、金额和业务归属映射一致。
3. V3 prod 中不存在迁入的 personal 正式业务记录。
4. V3 生成的学生月结只读取迁入的正式课时，未出现重复或遗漏。
5. 学费账单、老师工资、收入和支出均由 V3 生成且业务归属为青空进学塾。
6. Cash 请求只来源于 V3 已生成的收入 / 支出，不关联 V2 旧 linkage。
7. 按学生、老师、月份汇总的课时数、时长和金额通过对账。
8. 迁移脚本重复运行时新增数量为零，证明幂等。
9. V2 仍可查询 2026-06 以前和历史个人名义数据。

## 10. 上线切换顺序

1. 建立独立 staging / prod 环境和密钥，禁止复用 dev 数据库。
2. 在 staging 使用脱敏或受控副本完成至少一次全流程迁移演练。
3. 冻结 V2 的 2026-07+ 新下游生成动作，导出迁移源清单和校验摘要。
4. 备份 V2 来源与 V3 prod 空库状态。
5. 导入引用主数据和 2026-07+ 青空进学塾课时。
6. 完成数量、关联、金额和幂等校验。
7. 在 V3 生成 7 月月结、账单、工资及后续财务链路。
8. 切换日常运营入口到 V3，V2 转为历史辅助查询。
9. 保留迁移批次报告、异常清单、操作者和时间戳用于审计。

任何校验失败都应停止下游生成并回滚本迁移批次，不在 prod 中手工拼接半套业务链路。
